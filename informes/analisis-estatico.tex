\begin{abstract}
Este informe documenta la configuraci\'on y ejecuci\'on de herramientas de
an\'alisis est\'atico sobre el backend (Java/Spring Boot) y el frontend
(JavaScript vanilla) del proyecto. Todas las herramientas se corrieron
realmente contra el c\'odigo del repositorio; los reportes crudos
(\texttt{.txt}/\texttt{.xml}/\texttt{.json}) est\'an disponibles en
\texttt{docs/informes/} para verificaci\'on.
\end{abstract}

\subsection*{An\'alisis del hallazgo EI\_EXPOSE\_REP2 (\texttt{Cita.java:38})}

\begin{lstlisting}[language=java]
public Cita(Long pacienteId, Doctor doctor, LocalDateTime fechaHora,
            String motivo, String estado) {
    this.pacienteId = pacienteId;
    this.doctor = doctor;               // <- SpotBugs: EI_EXPOSE_REP2 aqui
    ...
}
\end{lstlisting}
Si quien llama al constructor conserva su propia referencia a \texttt{doctor}
y la modifica despu\'es, esa modificaci\'on se refleja tambi\'en dentro de la
\texttt{Cita} ya creada, rompiendo encapsulamiento. En este proyecto el
impacto pr\'actico es bajo (Hibernate gestiona el ciclo de vida de las
entidades), pero es un patr\'on a evitar en general.

\textbf{Nota importante (falso negativo):} SpotBugs + FindSecBugs, con
\texttt{effort=Max} y \texttt{threshold=Low} (la configuraci\'on m\'as
sensible disponible), \textbf{no detect\'o} la inyecci\'on SQL real en
\texttt{DoctorService.buscarPorEspecialidadInsegura} (concatenaci\'on de
strings hacia \texttt{entityManager.createNativeQuery}). Esa vulnerabilidad
solo se evidenci\'o con pruebas de integraci\'on dirigidas -- ver Hallazgo~4
de \texttt{analisis-owasp.pdf}. Se documenta expl\'icitamente porque es una
lecci\'on metodol\'ogica central del proyecto: \textbf{el an\'alisis
est\'atico no reemplaza las pruebas de seguridad dirigidas.}

\section{ESLint -- desglose}

\begin{lstlisting}
frontend/js/citas.js
  11:7  warning  'CitasModule' is assigned a value but never used

frontend/js/doctores.js
  10:7  warning  'DoctoresModule' is assigned a value but never used

frontend/js/historias.js
  10:7  warning  'HistoriasModule' is assigned a value but never used

frontend/js/pacientes.js
  11:7  warning  'PacientesModule' is assigned a value but never used

4 problems (0 errors, 4 warnings)
\end{lstlisting}

\textbf{An\'alisis:} los 4 warnings son \textbf{falsos positivos} propios de
la arquitectura del frontend (sin bundler ni m\'odulos ES): cada objeto
\texttt{XModule} se usa exclusivamente desde atributos \texttt{onclick="..."}
dentro de cadenas HTML generadas din\'amicamente
(ej. \texttt{onclick="PacientesModule.editarPaciente(\$\{p.id\})"}), algo que
ESLint no puede rastrear est\'aticamente porque no es una referencia directa
en el AST de JavaScript. No representan c\'odigo muerto real: eliminar esos
objetos rompe la aplicaci\'on. Se documentan igual porque son la salida real
del linter, y porque evidencian una limitaci\'on conocida de este estilo de
arquitectura (dificulta el an\'alisis est\'atico) -- otro argumento a favor de
migrar a m\'odulos ES si el proyecto creciera.

\section{An\'alisis de 5+ hallazgos relevantes (requerido por la r\'ubrica)}

\begin{enumerate}[leftmargin=*]
  \item \textbf{Checkstyle -- Indentation (553 casos):} el proyecto completo
    usa 4 espacios; adoptar Google Style (2 espacios) de forma retroactiva
    tocar\'ia pr\'acticamente cada archivo. Se recomienda decidir un est\'andar
    de equipo \emph{antes} de escribir m\'as c\'odigo, no despu\'es.
  \item \textbf{Checkstyle -- MissingJavadocMethod/Type (47 casos):} ninguna
    clase p\'ublica del backend tiene Javadoc. Priorizar los repositorios con
    \texttt{@Query} personalizado (ej.
    \texttt{PacienteRepository.buscarPorNombre}), donde un comentario
    explicando el comportamiento real de la query evitar\'ia la confusi\'on
    encontrada en este mismo proyecto (ver nota sobre el comentario enga\~noso
    de esa query en \texttt{analisis-owasp.pdf}).
  \item \textbf{SpotBugs -- EI\_EXPOSE\_REP2 en \texttt{HistoriaClinica}
    (\texttt{HistoriaClinica.java:38-39}):} expone directamente
    \texttt{Paciente} y \texttt{Doctor}, dos entidades que contienen datos
    personales/sanitarios. Aunque el riesgo de mutaci\'on accidental es bajo
    en este proyecto, es exactamente el tipo de hallazgo que en un sistema
    real con m\'as consumidores de la API se vuelve relevante.
  \item \textbf{SpotBugs -- falso negativo en la inyecci\'on SQL real:}
    documentado arriba. Es el hallazgo m\'etodol\'ogicamente m\'as importante
    de todo el an\'alisis est\'atico: demuestra que la cobertura de
    herramientas SAST gen\'ericas no es completa y debe complementarse con
    pruebas de seguridad expl\'icitas.
  \item \textbf{ESLint -- 0 errores reales, 4 falsos positivos
    estructurales:} el resultado "limpio" de ESLint \textbf{no} significa
    ausencia de problemas de seguridad en el frontend -- los hallazgos XSS
    reales (m\'ultiples usos de \texttt{innerHTML} sin escapar) no son
    detectables por \texttt{eslint:recommended} sin un plugin de seguridad
    espec\'ifico (ej. \texttt{eslint-plugin-no-unsanitized}), que no estaba
    configurado. Se documenta como \'area de mejora para el checklist de
    calidad del equipo.
  \item \textbf{Checkstyle -- CustomImportOrder (31 casos) en los 4
    repositorios:} el orden de imports incorrecto es c\'osm\'etico, pero su
    repetici\'on id\'entica en \texttt{PacienteRepository},
    \texttt{DoctorRepository}, \texttt{CitaRepository} y
    \texttt{HistoriaClinicaRepository} sugiere que se generaron a partir de
    la misma plantilla -- un buen candidato para una regla de
    \texttt{.editorconfig} + formateador autom\'atico en pre-commit.
\end{enumerate}
